# كتالوج جملة — عناية وتجميل (B2B Wholesale Catalog)

منصة ويب عربية (RTL) لكتالوج طلبات الجملة في مجال التجميل والعناية والصيدليات. يتصفّح التاجر المنتجات بالليرة السورية، يبني السلة، ويرسل الطلب عبر واتساب — دون بوابة دفع. تُدار الأسعار والمخزون والكتالوج من لوحة إدارة بالدولار الأمريكي.

---

## وصف المشروع

هذا المشروع **عرض تجريبي احترافي (Demo)** جاهز للتسليم والعرض على العميل، ويمكن توسيعه لاحقاً إلى إنتاج كامل. يجمع بين:

- **واجهة تسويقية** (صفحة هبوط) لشرح نموذج العمل B2B.
- **متجر إلكتروني** خفيف ومحسّن للجوال.
- **لوحة إدارة** لإدارة المنتجات، التصنيفات، الأسعار، الإعدادات، والاستيراد/التصدير عبر Excel.
- **حسابات تجار** (تسجيل/دخول) مع إمكانية الطلب كضيف.

الهوية البصرية: **Teal + أبيض** — هادئة، نظيفة، ومناسبة لقطاع التجميل والجملة.

## الميزات الرئيسية

### للتاجر (واجهة المتجر)

- صفحة هبوط عربية مع أقسام: المميزات، التصنيفات، كيف يعمل، منتجات مميزة.
- كتالوج `/shop` مع تصنيفات، بحث، وبطاقات منتجات.
- صفحة تفاصيل منتج مع **متغيرات** (حجم / لون / رائحة…) وإضافة دفعة للسلة.
- سلة مشتريات مع درج (Drawer) على الجوال وصفحة سلة كاملة.
- إتمام الطلب عبر **واتساب** برسالة عربية منسّقة.
- تسجيل دخول / إنشاء حساب تجاري (اختياري — الطلب كضيف متاح).
- عرض الأسعار للعميل **بالليرة السورية** بعد التحويل والتقريب.

### للإدارة

- دخول محمي (`ADMIN_USER` / `ADMIN_PASSWORD`).
- إدارة المنتجات (صور، وصف، سعر USD، مخزون، إخفاء/نشاط، سلوك نفاد المخزون).
- إدارة المتغيرات (اسم، مخزون، سعر USD اختياري، تفعيل/تعطيل).
- إدارة التصنيفات.
- **تسعير جماعي** بنسبة مئوية (كل المنتجات أو تصنيف) مع معاينة قبل التطبيق.
- إعدادات المتجر: سعر الصرف، التقريب، واتساب، اسم ووصف المتجر.
- **تصدير/استيراد Excel** للمنتجات والتصنيفات والمتغيرات مع التحقق وملخص النتائج.
- لوحة إحصائيات (منتجات، مخزون، إعدادات).

### تقنيات مساعدة

- تحويل تلقائي USD → SYP مع تقريب 500 أو 1000 ل.س.
- سلوك نفاد المخزون: إخفاء / تعطيل / عرض رمادي.
- صور منتجات مع بديل محلي عند فشل الرابط.

## التثبيت والإعداد

### 1. استنساخ المشروع وفتح المجلد

```bash
cd "مسار/المشروع"
```

### 2. تثبيت الاعتماديات

```bash
npm install
```

> يشغّل `postinstall` تلقائياً أمر `prisma generate` لتوليد عميل Prisma.

### 3. إعداد متغيرات البيئة

انسخ القالب وأنشئ ملف `.env` في جذر المشروع:

```bash
cp .env.example .env
```

عدّل القيم حسب الحاجة (انظر [متغيرات البيئة](#متغيرات-البيئة)).

### 4. تطبيق هجرات قاعدة البيانات (Migration)

**بيئة التطوير** (ينشئ/يحدّث SQLite ويسجّل migrations):

```bash
npx prisma migrate dev
```

**بيئة الإنتاج** على الخادم:

```bash
npx prisma migrate deploy
```

أو عبر السكربت:

```bash
npm run db:migrate
```

> `db:migrate` يشغّل `prisma migrate dev` — للإنتاج استخدم `migrate deploy` يدوياً.

### 5. تعبئة بيانات العرض التجريبي (Seed)

```bash
npx prisma db seed
```

أو:

```bash
npm run db:seed
```

يُنشئ: إعدادات المتجر، تصنيفات، منتجات (مع متغيرات)، حساب تاجر تجريبي، وبيانات جاهزة للعرض.

### 6. تشغيل خادم التطوير

```bash
npm run dev
```

افتح المتصفح: **http://localhost:3000**

### 7. بناء نسخة الإنتاج

```bash
npm run build
npm start
```

- `npm run build` — يبني Next.js (صفحات + API).
- `npm start` — يشغّل الخادم على المنفذ الافتراضي **3000** (أو `PORT` من البيئة).

## متغيرات البيئة

| المتغير | مطلوب | الوصف |
|---------|--------|--------|
| `DATABASE_URL` | نعم | مسار قاعدة SQLite. مثال تطوير: `file:./prisma/dev.db`. على الخادم استخدم مساراً **مطلقاً ودائماً** وقابلاً للكتابة. |
| `ADMIN_USER` | نعم (إنتاج) | اسم مستخدم لوحة الإدارة. الافتراضي: `admin`. |
| `ADMIN_PASSWORD` | نعم (إنتاج) | كلمة مرور الإدارة. الافتراضي: `demo123` — **غيّرها قبل الإنتاج.** |
| `SESSION_SECRET` | يُنصح به (إنتاج) | سر توقيع **جلسة التاجر**. في الكود اسم المتغير: `CUSTOMER_SESSION_SECRET`. إن لم يُضبط، يُستخدم `ADMIN_PASSWORD` كبديل (غير مثالي للإنتاج). |
| `PORT` | اختياري | منفذ التشغيل عند `npm start` (تضبطه لوحة الاستضافة أحياناً). |
| `NODE_ENV` | تلقائي | `production` عند التشغيل بـ `npm start`. |

مثال ملف `.env` للتطوير:

```env
DATABASE_URL="file:./prisma/dev.db"
ADMIN_USER="admin"
ADMIN_PASSWORD="demo123"
CUSTOMER_SESSION_SECRET="ضع-هنا-سلسلة-سرية-طويلة-عشوائية"
```

> **ملاحظة:** لا ترفع ملف `.env` إلى Git. استخدم `.env.example` كقالب فقط.

## منطق التسعير

### 1. السعر في الإدارة — USD

- يُدخل سعر المنتج (والمتغير اختيارياً) **بالدولار الأمريكي** في لوحة الإدارة.
- هذا هو «مصدر الحقيقة» للتسعير عند التعديل اليدوي أو عبر Excel أو التسعير الجماعي.

### 2. سعر الصرف

- من **إعدادات المتجر** (`/admin/settings`): حقل **USD → SYP** (مثال: `15000`).
- يُطبَّق على كل عرض سعر للعميل في المتجر والسلة والواتساب.

### 3. خطوة التقريب (Rounding)

- خياران: **أقرب 500 ل.س** أو **أقرب 1000 ل.س**.
- الصيغة (مبسّطة): `التقريب = round((السعر_بالدولار × سعر_الصرف) / الخطوة) × الخطوة`
- التنفيذ في: `lib/pricing.ts`

### 4. ما يراه العميل — SYP

- في المتجر تظهر أسعار جاهزة بالليرة (مثال: `450,000 ل.س`).
- السلة والواتساب يستخدمان نفس المنطق لضمان الاتساق.
- ملاحظة في رسالة الواتساب: الأسعار قابلة للتأكيد حسب التوفر وسعر الصرف النهائي.

**مثال:** منتج بـ `10 USD`، صرف `15000`، تقريب `1000` → `150,000 ل.س`.

## تدفق الطلب عبر واتساب

1. يضيف التاجر منتجات (ومتغيرات) إلى **السلة** من المتجر أو صفحة المنتج.
2. من السلة أو `/checkout` يُدخل: **الاسم، الهاتف، العنوان، ملاحظات** (واختيار طلب توصيل إن وُجد).
3. يمكن تعبئة البيانات تلقائياً إن كان مسجّل دخولاً أو فعّل «تذكّر بياناتي».
4. عند التأكيد يُبنى نص عربي منسّق (`lib/whatsapp.ts`) يتضمن:
   - بيانات العميل
   - كل سطر: اسم المنتج (+ المتغير)، الكمية، سعر القطعة، إجمالي السطر
   - الإجمالي النهائي بالليرة
   - ملاحظات العميل
5. يُفتح رابط `wa.me` برقم **واتساب الاستلام** من إعدادات المتجر.
6. **لا يوجد دفع إلكتروني** — التأكيد والتحصيل خارج المنصة عبر واتساب.

## استيراد وتصدير Excel

### التصدير

1. ادخل إلى **/admin/import-export**.
2. اضغط **تحميل products-export.xlsx**.
3. الملف يتضمن أوراق **Categories** و **Products** (صف لكل متغير إن وُجد).

### الاستيراد

1. عدّل الملف محلياً (احتفظ بأسماء الأعمدة: `productId`, `productNameAr`, `categorySlug`, `variantLabel`, `priceUsd`, `stockQty`, `isActive`).
2. من نفس الصفحة ارفع الملف `.xlsx`.
3. **التحقق ومعاينة** — يعرض عدد الأسطر وعينة؛ أي أخطاء في الأعمدة تظهر قبل التطبيق.
4. **تأكيد الاستيراد** — يحدّث المنتجات/المتغيرات الموجودة فقط (حسب `productId` و`variantLabel`).
5. ملخص النتائج: **جديد / محدّث / متخطى / أخطاء**.

> الاستيراد الحالي يركّز على **تحديث** سجلات موجودة؛ إنشاء منتجات جديدة بالكامل يتم من لوحة الإدارة أو بتوسيع لاحق.

## النشر على Hostinger واستضافة مشتركة

### هل يعمل على استضافة مشتركة عادية (ملفات ثابتة / PHP فقط)؟

**لا.** هذا التطبيق يحتاج:

- عملية **Node.js** تعمل باستمرار (`npm start`).
- مسارات **API** و **Prisma** وملف **SQLite** قابل للكتابة.
- **Middleware** لحماية الإدارة والحسابات.

رفع ملفات HTML إلى `public_html` **لا يشغّل** المتجر ولا الإدارة.

### متى يعمل على Hostinger؟

- إذا كانت خطتك تدعم **تطبيق Node.js** (حسب ما توفره Hostinger في لوحتك).
- اتبع **[PRODUCTION_CHECKLIST.md](./PRODUCTION_CHECKLIST.md)** لخطوات ما قبل/بعد النشر واختبار القبول.
- اتبع **[DEPLOYMENT.md](./DEPLOYMENT.md)** للخطوات التفصيلية بالعربية: المتغيرات، `migrate deploy`, البناء، النسخ الاحتياطي لـ SQLite.
- قالب إنتاج: **[`.env.production.example`](./.env.production.example)** — أوامر: `npm run deploy:checklist`, `npm run deploy:prepare`, `npm run start:prod`.

### بدائل إن لم يتوفر Node على Hostinger

| الخيار | ملاحظة |
|--------|--------|
| ترقية Hostinger / VPS | تحكم كامل + Node |
| Vercel + PostgreSQL (Supabase) | ممتاز لـ Next.js؛ انقل القاعدة من SQLite |
| Railway / Render | Node + قرص دائم أو Postgres |
| تحويل لاحق إلى PHP + MySQL | مشروع منفصل — ليس نشر هذا المستودع كما هو |

**غير موصى به:** تحويل الإدارة إلى `localStorage` فقط — يفقد الأمان والمزامنة وExcel.

## الدعم بعد التسليم

يُفترض تسليم المشروع مع:

| البند | التفاصيل |
|-------|----------|
| الكود المصدري | مستودع كامل مع `README.md` و `DEPLOYMENT.md` |
| التشغيل المحلي | تعليمات واضحة أعلاه + بيانات demo |
| قاعدة البيانات | SQLite + migrations + seed |
| التوثيق | هذا الملف + دليل النشر |

**نطاق الدعم المعتاد بعد التسليم** (يُحدد في العقد):

- شرح تشغيل محلي ونشر أول مرة.
- إصلاح أخطاء واضحة في الكود المُسلّم خلال فترة متفق عليها.
- استفسارات حول هيكل المشروع والمسارات.

**خارج النطاق الافتراضي** (ما لم يُذكر في العقد):

- استضافة وإدارة سيرفر العميل.
- إضافة بوابة دفع، تطبيق موبايل native، أو multi-tenant.
- تحويل كامل إلى PHP/MySQL دون اتفاق منفصل.

للتواصل الفني بعد التسليم: احتفظ بنسخة من `.env` الإنتاج، نسخ احتياطية لملف `.db`، وسجّل إصدار Node المستخدم على الخادم.

## أفكار توسعة مستقبلية

- **قاعدة بيانات إنتاج:** PostgreSQL (Supabase / Neon) بدل SQLite.
- **رفع صور:** تخزين S3 أو تخزين Hostinger بدل روابط URL.
- **عدة مستخدمين إدارة** بأدوار (مدير / محرر).
- **إشعارات:** بريد أو Telegram عند طلب جديد (بالإضافة لواتساب).
- **تقارير مبيعات** وتاريخ طلبات (حفظ الطلبات في DB).
- **لغات إضافية** أو عملات أخرى.
- **تطبيق PWA** للتثبيت على الشاشة الرئيسية.
- **ربط ERP** أو نظام محاسبة.
- **كوبونات وحد أدنى للطلب** للجملة.
- **هجرة Hostinger PHP + MySQL** مع إعادة استخدام منطق `lib/pricing` و `lib/whatsapp`.
