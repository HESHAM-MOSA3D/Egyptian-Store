# Prompt لـ Cursor — Demo منصة كتالوج طلبات جملة (تجميل وعناية شخصية)

> **الهدف:** بناء Demo عملي سريع يعرض كل الميزات الأساسية للعميل. الكود نظيف وقابل للتوسع. التصميم نهائي قريب من الإنتاج (ليس wireframe فقط).

---

## 1) سياق المشروع

أنشئ **منصة كتالوج ذكية لطلبات الجملة** (ليست متجر e-commerce تقليديًا) لمركز يتاجر بمواد التجميل والعناية الشخصية.

- **الجمهور:** زبائن محلات وصيدليات + إمكانية التوصيل.
- **الأولوية:** سرعة، سهولة، تقليل عبء الإدخال على العميل، تجربة ممتازة على الموبايل.
- **الهوية البصرية:** هادئة — **Teal (أخضر بترولي)** + **أبيض**، خطوط عربية واضحة (مثل: Cairo أو Tajawal).
- **العملة للعميل:** عرض الأسعار بالليرة السورية **SYP** فقط في الواجهة.
- **الإدخال الداخلي:** الأسعار في لوحة التحكم بالدولار **USD**، مع **سعر صرف قابل للتعديل** من الإعدادات، وحساب تلقائي للسعر بالليرة مع **تقريب نهائي** (انظر القسم 6).

**ملاحظة استضافة (للتخطيط):** الإنتاج النهائي على **Shared Hostinger** (بدون VPS). صمّم المعمارية بحيث يمكن نشرها لاحقًا على استضافة مشتركة (PHP + MySQL مفضل للإنتاج). للـ Demo يمكن التشغيل محليًا مع بيانات تجريبية، مع فصل واضح بين Front و Back.

## 2) التقنيات المقترحة (التزم بها أو اقترح بديلًا مع مبرر)

### Demo (تطوير سريع في Cursor)
- **Frontend:** Next.js 14+ (App Router) + TypeScript + Tailwind CSS
- **UI:** shadcn/ui أو مكوّنات مخصصة متسقة مع Teal/White
- **State:** React Context أو Zustand للسلة وبيانات العميل
- **Backend للـ Demo:** يمكن API Routes في Next.js + **SQLite** (Prisma) أو JSON mock — مع طبقة Repository قابلة للاستبدال بـ PHP/MySQL لاحقًا
- **i18n:** واجهة عربية RTL بالكامل (`dir="rtl"`)

### مسار الإنتاج (لا تكسره في التصميم)
- طبقة **REST API** نظيفة (Products, Categories, Variants, Orders, Settings, ExchangeRate)
- لا تربط المنطق بعمق بإطار واحد؛ ضع منطق التسعير والتقريب في **خدمة مستقلة** `pricing.service`

## 4) نموذج البيانات (Data Model)

### Category
- `id`, `nameAr`, `slug`, `sortOrder`, `isActive`

### Product
- `id`, `categoryId`, `nameAr`, `descriptionAr`, `images[]`, `priceUsd` (سعر أساسي إن لم يكن للمتغيرات)
- `stockTotal` (اختياري — أو يُحسب من المتغيرات)
- `isActive`, `isHidden`, `outOfStockBehavior`: `hide` | `disable` | `grayed`
- `hasVariants`: boolean

### ProductVariant (خيارات: لون / رائحة / نوع)
- `id`, `productId`, `labelAr` (مثل: فحم، نعناع، أحمر)
- `sku` (اختياري), `priceUsd` (override أو null لاستخدام سعر المنتج)
- `stockQty`
- `sortOrder`, `isActive`

### Settings (singleton)
- `usdToSypRate` (رقم عشري)
- `roundingMode`: `nearest_500` | `nearest_1000`
- `whatsappNumber` (مثل: +9639xxxxxxxx)
- `storeNameAr`, `deliveryNoteAr`

### CustomerProfile (محليًا في المتصفح + اختياري في DB للـ Demo)
- `name`, `phone`, `address`, `notes`, `lastUsedAt`

### Order (للـ Demo — تسجيل محلي أو DB)
- `id`, `customer`, `lines[]`, `totalSyp`, `createdAt`, `status`: draft | sent_whatsapp

## 5) واجهة الزبائن (Front-end) — متطلبات تفصيلية

### 5.1 الصفحة الرئيسية / الكتالوج
- شريط علوي: اسم المركز، بحث سريع، أيقونة السلة بعدد القطع.
- **تصنيفات** أفقية قابلة للتمرير: عناية بالبشرة، الشعر، العطور، الكريمات، مكياج، أدوات… (من الـ seed).
- عند اختيار تصنيف: شبكة **بطاقات منتجات** (صورة، اسم، سعر SYP بعد التقريب، شارة "نفد" إن لزم).
- **Skeleton loading** و **empty states** عربية.

### 5.2 بطاقة المنتج
- صورة رئيسية، اسم مختصر، أقل سعر SYP أو "من X ل.س".
- إذا `outOfStock` حسب السياسة:
  - `hide`: لا يظهر في القائمة
  - `disable`: يظهر معطّل ولا يُضاف للسلة
  - `grayed`: يظهر باهت + badge "غير متوفر"

### 5.3 صفحة تفاصيل المنتج
- معرض صور (swiper أو thumbnails).
- وصف كامل.
- **إذا hasVariants:**
  - جدول/بطاقات خيارات: كل صف = `labelAr` + حقل كمية (number stepper) + سعر SYP للوحدة + المتاح من المخزون.
  - زر **"إضافة الكل للسلة"**: يضيف فقط الخيارات التي كميتها > 0 دفعة واحدة.
- **إذا لا variants:** كمية واحدة + إضافة للسلة.
- منع تجاوز `stockQty` مع رسالة عربية واضحة.

### 5.4 السلة
- قائمة بالأسطر: منتج، خيار (إن وجد)، كمية، سعر وحدة SYP، المجموع.
- تعديل كمية / حذف.
- **المجموع الكلي** بالليرة السورية (مُقرب).
- زر "إتمام الطلب".

### 5.5 إتمام الطلب (Checkout خفيف)
- نموذج: الاسم، رقم الهاتف (تحقق بسيط)، العنوان، ملاحظات.
- **حفظ تلقائي** في `localStorage` (واسترجاع عند العودة).
- checkbox "التوصيل مطلوب" (اختياري).
- زر نهائي: **"إرسال الطلب عبر واتساب"**.

### 5.6 تكامل واتساب
- عند الإرسال: بناء رسالة **منظمة بالعربية** مثل:

```
🛒 طلب جديد — [اسم المتجر]
────────────────
👤 الاسم: ...
📞 الهاتف: ...
📍 العنوان: ...
📝 ملاحظات: ...

📦 تفاصيل الطلب:
1) [منتج] — [خيار] × [كمية] = [سعر السطر] ل.س
...
────────────────
💰 الإجمالي: [X] ل.س
🕐 [تاريخ ووقت]
```

- فتح `https://wa.me/{whatsappNumber}?text={encodedMessage}`
- رقم الواتساب يُقرأ من **Settings** (قابل للتغيير من لوحة التحكم).

### 5.7 الأداء وتجربة الموبايل
- Mobile-first، أزرار كبيرة، مسافات مريحة للمس.
- صور lazy-loaded، WebP إن أمكن.
- لا صفحات ثقيلة؛ التنقل سريع (< 3 نقرات للطلب).

## 6) منطق التسعير والتقريب (إلزامي — اختبره)

```text
priceSypRaw = priceUsd * usdToSypRate

إذا roundingMode === nearest_500:
  priceSyp = round(priceSypRaw / 500) * 500
إذا roundingMode === nearest_1000:
  priceSyp = round(priceSypRaw / 1000) * 1000
```

- اعرض الأسعار للزبون **بعد التقريب** دائمًا.
- في لوحة التحكم: أظهر **USD** + **SYP المحسوبة** live عند تغيير السعر أو سعر الصرف.
- عند تغيير سعر الصرف: أعد حساب عرض الأسعار في الواجهة (لا حاجة لتحديث يدوي لكل منتج).

**أمثلة اختبار في seed:**
- 10 USD × 15000 = 150000 → تقريب 1000 = 150000
- 10 USD × 15250 = 152500 → تقريب 500 = 152500 أو 1000 = 153000 (وضّح السلوك في UI)

## 7) لوحة التحكم (Dashboard)

### 7.1 دخول Admin (للـ Demo)
- مسار `/admin/login` — مستخدم/كلمة مرور من `.env` (admin / demo123) — لا تعقيد OAuth في الـ Demo.

### 7.2 إدارة التصنيفات
- CRUD: اسم، ترتيب، تفعيل/إيقاف.

### 7.3 إدارة المنتجات
- قائمة مع فلترة (تصنيف، نشط، نفد).
- نموذج إضافة/تعديل:
  - اسم، وصف، تصنيف، صور (رفع ملف أو URL للـ Demo)
  - سعر USD (للمنتج أو لكل variant)
  - كميات المخزون
  - تفعيل/إخفاء
  - سلوك نفاد المخزون: hide / disable / grayed
  - **محرر variants:** إضافة/حذف صفوف (labelAr, priceUsd, stockQty)

### 7.4 تعديل أسعار جماعي (Bulk)
- شاشة: نسبة مئوية (+10% / -5%) أو أكثر من نسبة متتالية.
- النطاق: **كل المنتجات** أو **تصنيف محدد** أو **منتجات محددة (checkbox)**.
- معاينة قبل التطبيق: عدد المنتجات المتأثرة.
- يطبق على `priceUsd` للمنتج والـ variants التي لها سعر مستقل.

### 7.5 الإعدادات
- سعر الصرف USD → SYP
- وضع التقريب (500 / 1000)
- رقم واتساب الطلبات
- اسم المتجر، نص توصيل افتراضي

### 7.6 استيراد / تصدير Excel
- **تصدير:** ملف `.xlsx` أعمدة مقترحة:
  - `productId`, `productName`, `category`, `variantLabel`, `priceUsd`, `stockQty`, `isActive`
- **استيراد:** رفع ملف → معاينة → تحذير أخطاء → تأكيد
- دعم تحديث الموجود وإضافة جديد (مطابقة بـ SKU أو productId+variantLabel)
- استخدم مكتبة `xlsx` أو `exceljs`

### 7.7 سلوك نفاد الكمية
- عند `stockQty <= 0` لكل variants المنتج: طبّق `outOfStockBehavior` تلقائيًا.
- في Admin: badge "نفد" + فلتر "نفد".

## 8) بيانات تجريبية (Seed) — مطلوبة للـ Demo

أنشئ **15–25 منتجًا** موزعة على 5+ تصنيفات، منها:

1. **صابون طبيعي** بvariants: فحم، نعناع، صبار — كميات مختلفة، بعضها نفد.
2. **عطر** بvariants روائح.
3. **كريم** بدون variants (كمية واحدة).
4. **شامبو** بvariants أحجام/أنواع.
5. منتج **مخفي** ومنتج **معطّل** ومنتج **رمادي** لعرض السياسات الثلاث.

سعر صرف افتراضي: `15000` — تقريب `1000`.

## 11) تعليمات التنفيذ لـ Cursor (Workflow)

1. **ابدأ** بقراءة هذا الملف كاملًا ثم أنشئ المشروع والـ schema.
2. **نفّذ** `pricing.service` + unit tests بسيطة للتقريب.
3. **ابنِ** API ثم واجهة الزبائن ثم `/admin`.
4. **أضف** seed غنيًا بالعربية.
5. **اختبر** يدويًا على عرض موبايل (Chrome DevTools).
6. **سلّم** README + لقطات أو فيديو GIF اختياري لمسار الطلب كامل.

**أسلوب الكود:** TypeScript strict، مكوّنات صغيرة، أسماء عربية للعرض فقط (الحقول `nameAr`)، تعليقات قليلة للمنطق غير الواضح.

**لا تستخدم** بيانات وهمية فقط في الواجهة بدون API — اربط كل شاشة بمصدر بيانات حقيقي (حتى لو SQLite).
